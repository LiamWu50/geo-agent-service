# 后端 geo-agent-service Chat 图层样式控制开发文档

## Problem Statement

前端已向 AI Chat 提供已加载图层的名称、ID、几何类型、可见性和透明度，但后端当前不会将“改颜色、调整点大小、加粗线条、修改面填充”等自然语言请求识别为地图样式操作，也不会发出可执行的样式命令。

现有后端只会为分析结果发送 `layer.addDataset`，为图层查看发送定位和标记命令。虽然前端协议中预留了显隐和透明度能力，生产代码也没有为用户样式请求生成这些命令。因此用户的制图意图最终只能变成普通文本回答，无法反映到地图。

## Solution

后端在 AI Chat 的地图命令编排中新增“图层样式控制”意图：依据前端传入的图层上下文、几何类型、可编辑能力和客户端命令能力，解析目标已加载图层与受限样式补丁，生成 SSE `map.command` 事件，命令动作为 `layer.updateStyle`。

第一阶段采用确定性优先的规则编排：仅在用户请求包含明确样式意图、能够唯一匹配一个可编辑矢量图层、且客户端声明支持该命令时生成命令。目标不唯一、缺少图层、属性超出支持范围时，后端返回澄清事件或可读说明，而不是猜测并修改地图。

## User Stories

1. 作为地图使用者，我想说“把机场图层改成红色”，以便 AI 直接修改正确的已加载图层。
2. 作为地图使用者，我想说“把港口线加粗到 4 像素”，以便突出运输通道。
3. 作为地图使用者，我想说“人口稠密区填充为半透明橙色”，以便表达专题含义。
4. 作为地图使用者，我想说“把当前图层的点放大”，以便在不清楚内部图层 ID 时完成操作。
5. 作为地图使用者，我想在同名或相似图层并存时得到选择问题，以便避免误修改。
6. 作为地图使用者，我想在请求影像调色、分级设色等本期不支持的能力时得到明确反馈，以便知道限制所在。
7. 作为地图使用者，我想在浏览生成的分析结果图层后继续修改其样式，以便让分析结果可读。
8. 作为前端调用方，我想让后端只发送客户端声明支持的地图命令，以便避免无效 SSE 事件。
9. 作为后端开发者，我想将样式命令与空间分析工具分开，以便视觉操作不会触发不必要的数据计算或生成新数据集。
10. 作为后端开发者，我想保存命令生成的原因和目标图层信息，以便审计聊天行为和排查问题。
11. 作为测试人员，我想通过 SSE 响应验证正确的图层样式命令被发出，以便覆盖真实 API 合约。
12. 作为测试人员，我想验证含糊、非法或不支持的样式请求不会产生 `map.command`，以便避免误操作。
13. 作为产品维护者，我想在未来为影像、3D Tiles 和专题渲染增加处理器，而不改变现有矢量样式命令的含义。

## Implementation Decisions

- 将 `layer.updateStyle` 作为新的地图命令动作加入 AI Chat 与前端共同使用的协议。后端事件数据直接放置命令对象，继续沿用现有 `map.command` SSE 封装，不再新增平行的“style event”。

- 后端只在前端 `clientCapabilities.mapCommands` 含有 `layer.updateStyle` 时生成该命令。客户端能力缺失时，返回文本说明或澄清，不发送前端无法执行的操作。

- 样式命令结构与前端保持一致：`action`、`layerId` 和受限 `style` 补丁。后端只生成 JSON 可序列化的声明式值，不生成 Cesium 类、脚本、表达式或自由格式 JSON。

- 图层上下文扩展为包含 `editable.style`、当前 `style` 摘要和几何类型。后端将 `layerId` 作为命令目标的唯一标识；`datasetId` 仅用于解析、数据摘要关联和审计，不作为最终修改目标。

- 新增样式意图识别与命令规划步骤，位于现有地图展示命令规划的同一工作流层。识别词覆盖颜色、点大小、点描边、线宽、线色、面填充、面边框、透明度等常见中文表达，并支持英文关键字作为兼容输入。

- 颜色解析使用受控词典和 CSS 颜色规范化。中文颜色词先映射为预定义十六进制色值；用户直接输入的十六进制色值和受支持的 `rgb/rgba` 形式经校验后透传。不能解析的颜色不生成命令，应要求用户补充。

- 样式字段与几何类型交叉校验：点图层只允许 `point` 补丁，线图层只允许 `line` 补丁，面图层只允许 `polygon` 补丁；混合图层仅在用户明确指定对象类型时生成相应补丁。字段与目标几何不匹配时输出说明而不是静默忽略。

- 图层解析规则按确定性优先执行：先匹配请求中明确的图层 ID 或完整名称，再匹配唯一的规范化名称，最后在“当前图层/选中图层”语义下使用唯一的活动可编辑图层。零个或多个候选均产生 `clarification` 事件，不产生样式命令。

- 纯样式请求不调用 `geoprocess`、`spatial_filter`、属性统计或数据写入工具，也不创建 `layer.created` 事件。它只产生 `map.command`、对应的状态消息和必要时的澄清事件。

- 后端记录样式命令到本次运行的 `map_commands` 和事件历史，保存目标 `layerId`、`datasetId`、规范化后的补丁和生成原因。样式本身不写回 GIS 数据集；前端是本期唯一的渲染状态所有者。

- 已存在的地图展示请求继续生成 `camera.flyTo` 和 `overlay.addMarker`；分析工具继续生成 `layer.addDataset`。样式意图与展示意图同时出现时，可按“先更新样式，再定位”的顺序发送两个命令。

- 对显隐和透明度保持兼容：明确的“隐藏/显示”仍生成既有 `layer.setVisible`；仅请求透明度且未包含其他样式属性时可继续生成 `layer.setOpacity`；样式补丁携带颜色等属性时，透明度作为颜色 alpha 纳入 `layer.updateStyle`。

## Testing Decisions

- 测试以 `/api/ai-chat/sessions/{sessionId}/messages` 的 SSE 响应为最高层切面。断言事件顺序和命令载荷，而不是断言内部意图识别函数或工作流节点的调用细节。

- 基础正向场景至少覆盖：点图层改颜色与大小、线图层改颜色与宽度、面图层改填充色与边框、生成结果图层再改样式、样式后定位。

- 目标解析场景覆盖：精确名称、唯一模糊名称、当前唯一活动图层、无匹配图层和多个候选图层。后两种必须验证只产生 `clarification` 或文本事件，而不产生 `map.command`。

- 输入校验场景覆盖：未知颜色、非法 CSS 颜色、负数或超大数值、颜色属性用于线图层、线宽用于点图层、空样式请求、客户端未声明能力。

- 能力协商场景验证：当前端不携带 `layer.updateStyle` 时，后端绝不发送该命令；当前端携带该能力时，响应中的动作必须在声明能力集合内。

- 回归测试覆盖既有数据摘要、图层检查、空间筛选、地理处理和地图展示请求，确保样式意图不会让原本的只读查看或分析问题误触发样式命令。

- 单元测试可以覆盖颜色规范化和图层候选排序，但应以 API 合约测试作为需求验收的主要依据。已有 AI Chat SSE 测试、地理处理工具测试和空间筛选工具测试是直接先例。

## Out of Scope

- 本期不持久化样式到数据库、数据集元数据、图层树或项目配置；刷新页面后的保留策略由前端后续需求决定。
- 本期不执行分级设色、唯一值渲染、属性驱动样式、标签规则、聚合渲染或 Cesium 表达式。
- 本期不支持影像、3D Tiles、GLTF、地形或底图样式控制。
- 本期不允许 Chat 直接执行任意前端脚本或传递 Cesium 专有对象。
- 本期不引入通用的多轮确认流程；只有目标图层不唯一或请求信息不足时才使用现有澄清事件。

## Further Notes

- 后端的职责是“理解意图并生成受限命令”，不是计算或保存渲染样式。实际样式应用和用户可见结果仍由前端负责。
- 前后端应把样式命令 schema 视为版本化契约。新增图层类型或样式属性时必须同步扩展客户端能力、前端校验器和后端生成器，并补充 API 合约测试。
- 当前 `MapLayerResult` 已存在可选 `style` 字段，但它适用于新建结果图层的描述，不能替代对已加载图层的 `layer.updateStyle` 命令。
